### 3.2 High

#### [AUDIT-perspective-O14] 特商法表記が placeholder のまま（有償 tip-jar 公開前必須）
- **カテゴリ**: #4 観点反映 / #9 法務
- **検出根拠**: tip-jar（固定 100 円単発支援、concept §1.1 UC#8）= 国内向け有償サービス提供 → O14 require:[国内向け,有償,公開] ヒット。`src/content/legal/sct.ts` は「販売者/連絡先/支払方法/引渡時期/返品条件 を公開前に整備します」の placeholder のみ。**ただし現状 Stripe 未 install + checkout/webhook は 501 stub** = 課金は未 live。
- **判定**: 課金を live 化する前に特商法表記の実値整備が必須（法令）。**Class C**（販売者名/住所 = 運営者本人しか持たない情報）。
- **推奨修正アクション**: 課金 live 化フェーズ（`/flow:release`）で `/flow:revise _shared/legal` により sct.ts に実値（販売者・住所・支払・引渡・返品）を記入。tip-jar を有効化しない限りブロックではない。
- **§8 登録**: 登録推奨（[論点] 化）。

#### [AUDIT-perspective-O22BE] ゲスト→アカウント連携 / サインアウトの UI 動線が無い（バックエンド実装済・UI 欠落）
- **カテゴリ**: #4 観点反映（O22 多部構成、step 3.6/3.10）
- **検出根拠**: バックエンドは `linkGuestToAccount`（src/lib/auth/account.ts）+ api/auth.ts link action 実装済だが、フロントに sign-in / account-link ボタン・signOut 動線が **0 件**（CF-20260609-009 / CF-20260617-004 の broad-match マスクパターン）。
- **判定（要設計判断）**: auth SPEC は連携トリガを「**課金/同期時**」と契約。だが (a) tip-jar は「ゲストのままワンタップ」= 課金は連携不要、(b) **複数端末同期は MVP UC として UI 化されていない** = 連携の MVP 入口が存在しない。→ 「連携基盤は実装済だが MVP に発火点が無い dormant 状態」。**concept §1.3 では auth が priority-2 ✅ 完了扱い**のため、SPEC が over-promise か、同期/連携を post-MVP に明示スコープアウトすべきかの **scope 判断が必要**（純粋な Class A auto-shoot ではない）。
- **推奨修正アクション**: 2 択を人間判断 — ①「複数端末同期/アカウント連携を MVP スコープに含める」→ `/flow:feature account`（連携 + サインアウト UI、両輪）／②「post-MVP にスコープアウト」→ `/flow:concept` UPDATE で auth を「ゲスト匿名のみ（連携は将来）」に明示し SPEC §7 スコープ外へ移動。**①の場合 E2E 検証のため signOut 動線（両輪）も必須**。
- **§8 登録**: 登録推奨。
